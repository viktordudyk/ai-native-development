В загальному можна сказати, що мої прогнози та очікування щодо моделей та їх можливостей суттєво не змінилися. Як я і говорив раніше, можливості архітектури трансформерів майже вичерпані, і зараз ми бачимо тільки кількісні покращення. Такі кількісні покращення не змінюють підходів до використання ШІ у розробці, описаних мною в попередніх статтях, але відчутно підвищують приріст продуктивності та комфорт від їх використання. Якщо ще влітку, навіть ті задачі, які добре вирішує ШІ, не завжди було вигідно йому доручати, через ненадійність та малу швидкість роботи - то зараз майже завжди доручення ШІ підходящої задачі приносить більше вигод, ніж втрат. Ідея, що ШІ - це основний інструмент розробника, зараз уже повністю підтвердилася (але не варто забувати про решту інструментів).

## Моделі

Тут описую свій досвід, з одного боку, це може бути упереджено, але враховуючи маркетинговий бум, практичний досвід часто значно корисніший за рекламні графіки та далекі від реальних задач бенчмарки. Варто відзначити, що усі останні моделі від топових гравців (OpenAI, Anthropic, Google) дуже близькі за можливостями, але все ж таки є певні сильні та слабкі сторони кожної моделі.

### 1. GPT-5.1 та GPT-5.1 Codex

Моделі від OpenAI. Перша більш загальна, друга більш спеціалізована на розробці.
Вельми універсальні, недорогі по токенам. Непогано показують себе в складних задачах.
Але є великий мінус - як і попередники GPT-5 та GPT-5 Codex доволі повільні, з нестабільною швидкістю роботи, що дуже знижує продуктивність та комфорт роботи з ними.
Можуть бути цікаві у випадках, коли основна модель не справилась, або результат викликає сумніви і потребує перевірки. Як дефолтну модель для розробки я б не рекомендував.

### 2. Claude Sonnet 4.5 та Claude Opus 4.5

Моделі від Anthropic, в принципі розроблені для задач програмування. Sonnet 4.5 більш дешева і підходить для простих задач. Opus 4.5 більш дорога і підходить для складних задач. Але у версії 4.5 розбіжності між ними не такі великі як у попередніх версіях, по ціні Opus 4.5 буває більш дешевим на складних задачах через те, що знаходить рішення за меншу кількість токенів. Тому і Opus, і Sonnet можуть стати вашими дефолтними моделями в залежності від ваших уподобань, а можете їх змінювати в залежності від задачі. Моделі швидкі, та дещо багатослівні, але це можна контролювати за допомогою prompt engineering. В порівнянні з іншими лідерами певно будуть найдорожчі, особливо на простих задачах, де якщо уже звикли до моделей Anthropic - можна використовувати Haiku 4.5 для економії. Але в будь-якому випадку, своїх грошей вони варті і є гарним вибором за замовчуванням для розробки.

### 3. Gemini 3 Pro

Модель від Google. Не є спеціалізованою для задач програмування, але має гарні можливості для розробки. Модель по швидкості приблизно рівна Claude Sonnet 4.5, але більш стабільна у цій швидкодії, що приємно для роботи в інтерактивному режимі. Дуже великим плюсом моделі є комбінація її слухняності, мінімуму самодіяльності та стабільної немалої швидкості роботи. Це особливо цінно, коли розробник використовує модель для інтерактивної роботи над малими підзадачами - а саме цей режим роботи є найбільш продуктивним та не погіршує якість коду. Крім того, модель чудово працює з інструментами, наприклад браузером, гарно емулює просторову уяву, що робить її, мабуть, найкращою для фронтенд розробки. Та і для бекенду модель теж є гарним кандидатом на роль дефолтної, і вибір між нею та моделями Anthropic є справою смаку.

## Інструменти

Тут мало що змінилося, але ще раз опишу основні моменти, можливо, цікаво буде для тих, хто зараз обирає.

### 1. GitHub Copilot

Все ще відстає від лідерів по можливостям та швидкодії. Але все ще лідирує в легкості використання та адаптації до вашого стеку, так як присутній для більшості популярних IDE в майже повному вигляді. Найкраще працює у VS Code, але й в інших IDE обмеження та проблеми незначні. Отримав також свій CLI, що дозволяє користуватись без IDE. Лишається оптимальним вибором для тих, хто не бажає змінювати звичні інструменти розробки. Але треба не забувати про привязку до GitHub та пов'язані з нею обмеження (описано в попередній статті).

### 2. Cursor

Все ще лідирує в можливостях та швидкодії. Має найкращий автокомпліт, який швидкий та вельми якісний. І все ще має певні проблеми та обмеження для деяких стеків розробки, які залежать від ліцензійно захищених компонентів і розширень для VS Code. Значно зросли можливості для роботи з асинхронними агентами, є зручний інтерфейс для їх контролю. Нові більш потужні моделі збільшують спектр ефективного застосування підходу з асинхронними агентами, але поки задач де це реально виправдане ще небагато.

### 3. Windsurf

Також чудовий варіант для переходу, якщо розробник використовує IDE з підтримкою плагіну Windsurf. Так як розробник має можливість використовувати розширення для старту, яке надає базові можливості у його звичну IDE, а паралельно, по можливості, без втрат продуктивності, освоювати окрему IDE Windsurf (fork VS Code, як і Cursor, з тими самими проблемами). Цікавий орієнтацією саме на агентний підхід, з мінімізацією ручного писання коду, що з розвитком якості і швидкодії моделей стає все більш актуальним.

### 4. Claude Code від Anthropic чи Codex Cli від OpenAI чи Gemini CLI від Google

Чисто консольні інструменти, заточені під моделі свого вендора. Отримують першими багато цікавих можливостей, але все таки самостійно годяться лише для агентної розробки. А банальний ШІ автокомпліт буває часто більш оптимальним для багатьох задач. Як основний інструмент все ще не можу рекомендувати, але якщо є можливість спробувати як доповнення до інших інструментів - це може бути корисно, так як агентний підхід до розробки стає все більш дієздатним.

### 5. Google Antigravity

IDE на базі VS Code, з великими агентними можливостями. Новинка від Google, підходить для розробки з різним рівнем автономності агентів. Поки лише прев'ю, але виглядає дуже схожим на Cursor в його сучасному вигляді як за ідеєю так і можливостями. Якщо Google зможе забезпечити кращу підтримку плагінів для VS Code, розробити свої чи домовитись з Microsoft, то це може стати дуже цікавим варіантом для розробки. Поки спостерігаємо та граємось, на реальні проекти рано.

_І наостанок, не забуваємо, що хоч ШІ дуже гарно росте кількісно, якісно він поки стоїть на місці. Тому, незважаючи на весь хайп та маркетинг, не забувайте про решту інструментів розробника, а головне за розум та навички розробника!_
