# Writing docs in this repo

This repo is an AI SDLC consulting practice. The docs are written for engineers (principal-dev level) and for client conversations. They get read by humans who can spot AI-generated text from across the room and will throw it out if it looks autogenerated.

## The audience

- Senior engineers, tech leads, principal devs on the client side.
- Sometimes the user's own team.
- Never marketing, never execs without a tech background.

If you catch yourself writing for "the business" or "stakeholders", stop. Wrong audience.

## House writing style (this is the hard rule)

The user manually checks for AI tells and runs automated detectors. The docs have to pass both. The full feedback memory is at [[feedback-writing-style]], but the short version:

1. **No em-dashes (—). Ever.** Use commas, parentheses, colons, periods, or restructure the sentence. The em-dash is the single biggest tell.
2. **No British spelling.** "behavior" not "behaviour", "organize" not "organise", "color" not "colour", "analyze" not "analyse". The user is in a US/global eng market and British spelling reads as AI.
3. **No "(X — definition of X)" inline glosses for jargon.** The audience knows what HITL, SOW, MCP, CI, T&M, CoE, SDD mean. Explaining them in parentheses screams AI. Drop the explanation. Trust the reader.
4. **Cut hard.** AI overproduces. After a first draft, cut 30 to 50 percent. Short sentences. Fragments are fine.
5. **Vary the rhythm.** Don't make every bullet the same length. Don't make every section follow the same template. Real humans write inconsistently.
6. **Dev language, not corporate.** "yells when X breaks" not "raises an exception when X fails". "half-built" not "in development". "cheat sheet" not "matrix". "babysit" not "supervise". "tanks" not "degrades significantly". Slack energy, not whitepaper energy.
7. **No moralizing, no hedging, no trailing summaries.** Don't write "it's important to note", don't summarize what you just said.
8. **No emojis** unless the user explicitly asks.

If asked to "humanize" or "fix AI slop" in an existing doc, rewrite aggressively. Don't just delete em-dashes and call it done. The whole rhythm and word choice has to change.

## Where things live

- `docs/` — long-form articles, research, frameworks (English by default).
  - `docs/sales-research/` — top-10-question research for client calls.
  - `docs/ua/` — Ukrainian translations and originals.
  - `docs/_sources/` — raw transcripts kept for citation, not for daily reading.
- `results/` — finished outputs from research or analysis.
- `promts/` — prompt templates (yes, spelled "promts" in the repo, don't rename it without asking).
- `integrations/` — integration notes for specific tools.
- `pptx/` — slide content.
- `README.md` at root.

When creating a new doc, put it in the directory that matches its purpose. Don't invent new directories.

## Language

- English by default for technical docs.
- Ukrainian is used too. If the user writes to you in Ukrainian, they may want the output in Ukrainian. If the file is in `to_ask/ua/` or has a Cyrillic filename, it's Ukrainian.
- Don't auto-translate between languages unless asked.
- When writing Ukrainian, the same anti-AI-slop rules apply: short sentences, no over-formality, no тире (em-dash), dev language.

## Filenames

- kebab-case for new files: `top-10-questions.md`, `spec-driven-development.md`.
- Don't rename existing files (even ones with typos like `promts`) without asking.
- `.md` for everything unless there's a reason to use `.txt`.

## Structure conventions

- H1 for the doc title, H2 for major sections, H3 sparingly.
- `---` horizontal rules between major sections are fine and the user uses them.
- Bold the question/heading inside numbered items: `**1. Question text?**` followed by the body. See `to_ask/top-10-questions.md` for the pattern.
- Don't add a "Context:" or "TL;DR:" block unless the doc actually needs one. Most short docs don't.

## What NOT to do

- Don't add a "Conclusion" section that just restates what was said.
- Don't add tables of contents to short docs.
- Don't add disclaimers or "this is a living document" notes.
- Don't include generation timestamps or "last updated" lines unless the user asks.
- Don't reference Claude, AI assistance, or the conversation that produced the doc.

## Sanity check before saving

Before writing the final version, scan for:

- Any `—` characters. Replace all of them.
- Any British spellings. Switch to American.
- Any `(X — short definition)` patterns. Delete the gloss.
- Any sentence longer than about 25 words. Split it.
- Any section that reads like every other section in structure. Vary one of them.
- Any "Conclusion", "Summary", "Overview", or "Introduction" headings on a short doc. Probably delete.
