# Q1: AI SDLC Practice Packages — Sell-Readiness & Selection

This question is **internal** — the answer must come from our Practice Lead / SDD champions, not from external research. This file is a framework for getting a clean answer.

---

## What to ask the Practice Lead

### 1. Sell-readiness status per package

For each of the seven AI SDLC Practice packages (Assessment, AI-Assisted Introduction, AI-Assisted Uplift, Agentic Engineering Intro, Module-based Agentic SDLC, AI-Native SDLC Transition, Brownfield to AI-Ready), capture:

| Field | Why it matters |
|-------|----------------|
| **Delivery status** | Ready / Partial / Concept-only |
| **Delivered engagements to date** | Evidence that this package shipped |
| **Named delivery lead** | Who do I bring on a client call |
| **Reference artifacts** | What I can show — slide deck, sample report, sanitized case study |
| **Known gaps** | What's missing that I should not promise on a first call |

A clean answer: a single matrix with one row per package and these five columns filled in.

### 2. Selection criteria — client → package

Ask for a short diagnostic — a flowchart or decision tree, not a long doc. The dimensions that usually matter:

- **AI-tooling maturity** (no use → ad-hoc Copilot → structured agentic workflows)
- **Codebase posture** (greenfield → modern brownfield → legacy)
- **Team size and stack** (single team / multi-team / enterprise; iOS / Android / Web / Backend / Data)
- **Primary outcome the client is buying** (velocity, quality, compliance, hiring leverage, cost)
- **Regulatory profile** (medtech / fintech / critical infrastructure vs. consumer)

A clean answer: a one-page diagnostic with 4-6 questions on the left and a recommended package on the right.

### 3. Boundary cases

These are the questions that usually come up on the first technical call:

- What is the **minimum viable engagement** we will sell? (Standalone Assessment? Single-team pilot only? Anything below scope X we refuse?)
- What is the **upper bound** — at what point do we hand off from AI SDLC Practice to a different practice (AI Business Transformation, AI in Product Engineering)?
- How do we handle a **client who wants only one workstream** (e.g., spec authoring training, but not the agent rollout)?
- What's the **default duration** per package — fixed weeks or flexible?

### 4. Co-sell with other AI Compass practices

AI Compass has four practices (AI SDLC, AI Guardrails, AI Business Transformation, AI in Product Engineering). On a first call:

- Which scenarios trigger a **co-sell** — e.g., regulated industries pull in AI Guardrails automatically?
- When do we explicitly **decline co-sell** to keep scope tight?

---

## Output format for the team's answer

Ask for two artifacts:

1. **Package readiness matrix** (one row per package × five columns above)
2. **Selection diagnostic** (one-pager with 4-6 client questions → recommended package)

Until both exist, every first-call conversation will improvise scope, and proposals will be inconsistent across the sales team.

---

## What NOT to accept as an answer

- *"All seven are available"* without delivery evidence per package
- *"Just call me and we'll figure it out"* — no diagnostic = no scalable sales motion
- *"It depends on the client"* without listing what it depends on

If the answer is "we are still building these packages," that is also a valid answer — but then we need to know which package is the safe default to lead with, and what we explicitly cannot commit to yet.
