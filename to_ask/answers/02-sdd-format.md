# Q2: SDD Specification Format & Spec/Code Lifecycle

SDD in 2025-2026 has converged on a shape: human-readable Markdown artifacts, split across requirements / design / tasks, living in the source repo and consumed by AI agents. Disagreement is on syntax (prose vs. EARS vs. a structured DSL), on whether the spec leads the code or trails it, and on how drift is policed. No industry-standard validator exists yet — CI enforcement is still DIY/community-plugin territory.

## Format options

| Approach | What it is | Pros | Cons |
|---|---|---|---|
| **EARS** (Mavin/Rolls-Royce, 2009) | 5 sentence templates — Ubiquitous, Event (`When…`), State (`While…`), Optional (`Where…`), Unwanted (`If…then…`) — each ends `the system shall…` | Unambiguous, LLM-parses cleanly, decades of aerospace use | No native slot for NFRs; verbose for CRUD; not a file format on its own |
| **Markdown templates** (Spec Kit, Kiro) | `spec.md` + `design.md` + `tasks.md` per feature, generated by slash commands (`/speckit.specify`, `/plan`, `/tasks`, `/implement`) | Diff-friendly in PRs, agent-agnostic (Copilot/Claude Code/Cursor/Gemini), low ceremony | Loose schema — drifts into prose; NFRs bolted on, not first-class |
| **Structured DSL + deltas** (OpenSpec) | Enforced headers `## ADDED/MODIFIED/REMOVED Requirements`; each `### Requirement:` needs a `#### Scenario:`; EARS used inside | Brownfield-friendly deltas, CLI validates structure, three-phase state machine (proposal → apply → archive) | Steeper learning curve; 1.x in 2026, tooling still maturing |
| **Constitution + spec** (Spec Kit) | `.specify/memory/constitution.md` of non-negotiables plus per-feature specs | Captures security/a11y baseline once, not per ticket | Constitution itself drifts; no built-in linter |

## How specs live with code

Consensus: **same repo, feature-branched, co-located with code**.

- **Spec Kit**: `specs/NNN-feature-name/` (auto-numbered, semantic branch) containing `spec.md`, `plan.md`, `research.md`, `data-model.md`, `contracts/`, `quickstart.md`, `tasks.md`. Constitution at `.specify/memory/constitution.md`.
- **OpenSpec**: `openspec/changes/<kebab-name>/` for in-flight work, archived to `openspec/changes/archive/YYYY-MM-DD-<name>/` on merge. Long-lived capability specs at `openspec/specs/<capability>/spec.md`.
- **Kiro**: per-feature `requirements.md` / `design.md` / `tasks.md`; repo layout is project-defined (docs thin here).

Spec lands in the same PR as the code it generated, reviewed together. Archiving (OpenSpec) prevents the `changes/` folder becoming a graveyard.

## Drift prevention (still immature)

No first-party CI validator ships in any major toolkit. What teams actually deploy:
1. **Contract tests** generated from `contracts/` (OpenAPI/AsyncAPI) — fail build on schema divergence.
2. **Community plugins** for Spec Kit: *CI Guard* (block merge on missing/drifted spec), *Spec Sync*, *Verify*. Unofficial.
3. **Pre-commit hooks** that re-run the agent's `/analyze` on changed paths.
4. **PR convention**: any code PR must touch a spec, or carry a `spec-exempt` label reviewed by a principal.

## Ambiguity, edge cases, NFRs

- **Ambiguity**: `/speckit.clarify` forces agents to surface unknowns before planning; OpenSpec's proposal phase does the same.
- **Edge cases**: EARS *Unwanted behaviour* (`If <trigger>, then the system shall…`) is the cleanest pattern; pair with `#### Scenario:` blocks for executable examples.
- **NFRs**: No standard slot. Practical move — put measurable NFRs (p95 latency, WCAG 2.1 AA, OWASP ASVS level, RPO/RTO) in the constitution as global invariants, then reference them per-feature with concrete thresholds. Avoid "must be fast."

## What to ask the team

1. One format per engagement (Spec Kit / OpenSpec / Kiro), or client-choice? Mixed shops pay a context tax.
2. Who owns the **constitution**, and how is it versioned across client projects?
3. What's our **drift-detection SLA** until toolkits ship validators — manual review, contract tests, or a custom Skill?
4. How do we author **NFRs** so they're testable, not aspirational?
