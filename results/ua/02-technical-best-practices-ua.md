# AI-Native розробка: Збірник технічних найкращих практик

**Мета:** Практичний, підкріплений доказами довідник для інженерних команд, які впроваджують AI-native практики розробки.

**Аудиторія:** Старші розробники, техліди, архітектори — люди, які хочуть патерни, які можна спробувати вже сьогодні.

**Станом на:** Березень 2026

---

## 1. Початок роботи — Перші 30 хвилин

### Оберіть свій інструмент

| Якщо ви...                              | Почніть з                   | Чому                                                      |
|----------------------------------------|-----------------------------|----------------------------------------------------------|
| У команді, яка вже використовує VS Code | **GitHub Copilot**          | Нульове тертя — нативна інтеграція, охоплює inline + agent mode |
| Оптимізуєте під найкращий агентний досвід | **Cursor**                  | Найкращий цикл план→редагування→перевірка, широкий вибір моделей, швидке автодоповнення |
| Комфортно працюєте в терміналі / потрібне використання в CI | **Claude Code (CLI)**       | Незалежний від IDE, працює віддалено, отримує фічі першим, найсильніший консольний агент |
| Досліджуєте з мінімальними зобов'язаннями | **Copilot у VS Code**      | Доступний безкоштовний рівень, не потрібно змінювати IDE   |

### Перше продуктивне завдання (зробіть це зараз)

Оберіть одну з трьох швидких перемог — кожна займає менше 10 хвилин:

**Швидка перемога 1: Згенерувати тести**
1. Відкрийте файл з непротестованою функцією.
2. Відкрийте панель агента (Copilot: `Cmd+I` / Cursor: `Cmd+L`).
3. Введіть: `Write unit tests for [function name]. Cover happy path, one negative case, and one edge case.`
4. Перегляньте. Прийміть те, що правильно, виправте те, що ні.

**Швидка перемога 2: Згенерувати документацію**
1. Відкрийте модуль або клас, який не має документації.
2. Промпт: `Generate JSDoc/docstring for every public method in this file. Include parameter descriptions, return types, and one usage example per method.`
3. Перегляньте. Тепер у вас є документація, яка вручну зайняла б 30+ хвилин.

**Швидка перемога 3: Рефакторинг з поясненням**
1. Відкрийте файл зі складною функцією (50+ рядків, вкладені умови).
2. Промпт: `Refactor this function into smaller, well-named functions. Explain each extraction decision.`
3. Перегляньте diff ТА пояснення. Агент навчає вас своєму ходу думок.

**Очікуваний результат:** 3-5 робочих тестів менш ніж за 5 хвилин. Модель знає ваш фреймворк — вона використає той test runner, який ваш проєкт вже імпортує.

**Чого НЕ варто очікувати:** Тести не будуть ідеальними. Деякі assertions можуть бути наївними, моки можуть бути неповними. Це нормально — ви інженер, агент — це прискорювач.

### Зміна мислення

Перестаньте думати: *«Напиши код за мене.»*
Почніть думати: *«Виконай це чітко специфіковане завдання і перевір результат.»*

Якість вашого результату прямо пропорційна точності вашого вхідного завдання. Це єдиний найважливіший принцип у AI-native розробці.

## 2. Specification-Driven Development (SDD) на практиці

### Ключове розуміння

> **Код більше не є головним артефактом — специфікація є.**

У AI-native робочому процесі точна специфікація є миттєво виконуваною. Специфікація стає основним джерелом істини; для чітко специфікованих завдань код стає значною мірою відтворюваним зі специфікації. Якщо ваша специфікація правильна, код (майже) безкоштовний.

### Правило 50/50

Інвестуйте ~50% часу завдання в специфікацію → отримайте значне прискорення у реалізації (ранні впроваджувачі повідомляють про 2-5x для типових завдань, з вищими множниками для обмежених, повторюваних задач).

Це звучить контрінтуїтивно. Але зверніть увагу: більшість часу на дебагінг виникає через неоднозначні вимоги. Коли план точний, сучасні моделі виконують його за секунди. Ви обмінюєте години дебагінгу на хвилини планування.

### 4-фазний життєвий цикл SDD

```
Description → Decomposition → Validation → Living Updates
```

1. **Description** — Опишіть, що система повинна робити, природною мовою зі структурованим синтаксисом.
2. **Decomposition** — Розбийте на атомарні одиниці, кожна з яких вміщується в контекст моделі (1k–3k рядків).
3. **Validation** — Визначте критерії прийняття, виведіть тест-кейси зі специфікації.
4. **Living updates** — У міру навчання оновлюйте специфікацію. Специфікація еволюціонує разом із системою.

### Синтаксис EARS — одна з ефективних мов специфікацій

**EARS** (Easy Approach to Requirements Syntax), спочатку розроблений для інженерії вимог в автомобільній/аерокосмічній галузі, забезпечує однозначні вимоги, які безпосередньо відображаються на тест-кейси. EARS перейшов з нішевого до масового використання у AI-native робочих процесах — Amazon Kiro IDE використовує специфікації у стилі EARS як свій рідний формат, а GitHub Spec-Kit прийняв структуровані шаблони вимог, безпосередньо натхненні патернами EARS. Хоча існують альтернативи (Given-When-Then, RFC 2119 keywords), EARS став де-факто стандартом для специфікацій, придатних для споживання агентами:

| Патерн | Шаблон | Приклад |
|---------|----------|---------|
| **Ubiquitous** | The system shall [action] | The system shall encrypt all PII at rest using AES-256 |
| **Event-driven** | When [event], the system shall [action] | When a user submits a payment, the system shall validate the card via Stripe API |
| **State-driven** | While [state], the system shall [action] | While the circuit breaker is open, the system shall return cached responses |
| **Optional** | Where [condition], the system shall [action] | Where the user has MFA enabled, the system shall require a second factor |
| **Unwanted** | If [condition], the system shall [action] | If the request payload exceeds 10MB, the system shall reject with 413 |

Кожна вимога EARS відображається 1:1 на property-based тест. Саме так SDD усуває неоднозначність.

### Ієрархія документів

| Документ | Автор | Споживач | Призначення |
|----------|--------|----------|---------|
| **PRD** | PM / стейкхолдер | Люди | Що ми будуємо і чому |
| **Technical Design** | Архітектор / старший інженер | Інженери | Системні рішення, безпека, масштабованість |
| **AI Execution Spec** | Інженер | AI-агент | Точний контракт на виконання (EARS + Mermaid-діаграми) |

**Зберігання:** Усі специфікації зберігаються в Git як **Markdown + Mermaid-діаграми**. Ніколи не зберігайте специфікації як зображення (DrawIO, PNG) — вони роздувають контекст і невидимі для агента.

### До/після: побудова сервісу сповіщень

**Без SDD (vibe coding):**
```
Prompt: "Build a notification service that sends emails and push notifications"
```
Результат: Агент генерує ~400 рядків. Половина граничних випадків відсутня. Email-шаблони захардкоджені. Немає логіки повторних спроб. Ви витрачаєте 3 години на дебагінг та латання.

**З SDD:**
```markdown
## Notification Service — AI Execution Spec

### Requirements (EARS)
- When a notification event is received, the system shall route to the 
  appropriate channel (email, push, SMS) based on user preferences
- Where the delivery attempt fails, the system shall retry with exponential 
  backoff (max 3 retries, base delay 1s)
- If all retry attempts are exhausted, the system shall log the failure 
  and emit a `notification.failed` domain event
- While the rate limiter is active for a channel, the system shall queue 
  notifications and process them when the window resets

### Interfaces
- Input: `NotificationEvent { userId, type, payload, channels[] }`
- Output: `DeliveryResult { channelResults: Map<Channel, Status> }`

### Constraints
- Email: use SendGrid SDK, templates stored in `/templates/{type}.hbs`
- Push: use Firebase Cloud Messaging
- Max payload: 4KB per notification
```
Результат: Агент генерує ~600 рядків з логікою повторних спроб, маршрутизацією каналів, обробкою помилок та відповідними тестами. Час на ревʼю: 20 хвилин. Загальна економія часу: ~2,5 години.

## 3. Контекстний інжиніринг — Навичка, яка вирішує все

### Золоте правило

> **Якщо знання не записане в кодовій базі або Markdown-файлі, для агента його не існує.**

Племінні знання вашої команди, конвенції зі Slack, незаписані патерни іменування — нічого з цього не потрапляє до моделі. Кожне архітектурне рішення, конвенція іменування та патерн інтеграції повинні існувати як текст у репозиторії.

### Файли конвенцій проєкту

Ці файли спрямовують поведінку агента по всьому вашому проєкту:

| Файл | Інструмент | Що туди записувати |
|------|------|--------------------|
| `.github/copilot-instructions.md` | GitHub Copilot | Загальнопроєктні конвенції, патерни іменування, бажані бібліотеки |
| `.cursorrules` | Cursor | Те саме призначення, формат специфічний для Cursor |
| `AGENTS.md` | Claude Code | Конвенції репозиторію, команди збірки, патерни тестування |
| `.instructions.md` (для кожної папки) | Copilot/Cursor | Конвенції для конкретної папки (напр., «всі файли в `/api` використовують патерн Express middleware») |

**Приклад `.github/copilot-instructions.md`:**
```markdown
## Project conventions
- Language: TypeScript 5.x, strict mode
- Framework: NestJS with CQRS pattern
- Tests: Vitest, co-located with source files (`*.spec.ts`)
- Database: PostgreSQL via Prisma ORM
- Error handling: Use Result<T, E> pattern, never throw from domain layer
- Naming: PascalCase for classes, camelCase for functions, SCREAMING_SNAKE for constants
- API responses: Always wrap in { data, error, meta } envelope
```

### Якість контексту > кількість контексту

**Неправильно:** Скинути весь репозиторій у контекст.
**Правильно:** Подавати свіжі, стислі, релевантні фрагменти.

Практичні техніки:
- **Інкрементальне розкриття** — Не завантажуйте все наперед. Нехай агент знаходить файли за потребою через свої інструменти.
- **Паралельний багатоагентний прохід** — Для великих завдань використовуйте окремих агентів для аналізу, планування та верифікації. Кожен отримує фокусований фрагмент контексту замість одного перевантаженого вікна.
- **Патерн «друге мозок»** — Доменні знання (архітектурні рішення, специфікації інтеграцій, особливості API вендорів) зберігаються як markdown у папці `/docs` або `/specs`, індексовані для пошуку.

### Skills та кастомізація агентів

Крім файлів конвенцій, сучасні агенти підтримують **Skills** — багаторазові, спільні пакети конфігурації агентів, які кодують командно-специфічні робочі процеси:

| Концепція | Що робить | Приклад |
|---------|-------------|--------|
| **Skills** (Anthropic) | Пакетовані інструкції + доступ до інструментів для конкретного типу завдань | «Frontend component skill», який завжди перевіряє Figma, використовує токени дизайн-системи, запускає візуальну регресію |
| **Custom agent modes** | Конфігурації агентів, налаштовані для різних робочих процесів | «Review mode» (лише читання, знаходить проблеми) vs «Implement mode» (пише код, запускає тести) |
| **Hooks** | Pre/post дії, що спрацьовують за подіями агента | Автоматичний запуск лінтера після кожного редагування файлу; автоформатування перед комітом |

Skills накопичуються з часом — колекція skills команди стає їхнім AI-плейбуком, який можна переносити на нові проєкти та нових членів команди.

### Розширене мислення для складних рішень

Сучасні frontier-моделі підтримують **extended thinking / reasoning modes** — довшу внутрішню deliberацію перед відповіддю:

| Випадок використання | Режим | Чому |
|----------|------|-----|
| Архітектурні рішення | Extended thinking УВІМКНЕНО | Агент повинен зважити компроміси, врахувати вплив на всю систему |
| Дебагінг складних проблем | Extended thinking УВІМКНЕНО | Довші ланцюги міркувань знаходять кореневі причини, а не симптоми |
| Рутинна реалізація | Стандартний режим | Швидкість важливіша за глибину для чітко специфікованих завдань |
| Код-ревʼю | Extended thinking УВІМКНЕНО | Глибший аналіз виявляє тонкі логічні помилки |

**Практична порада:** Увімкніть extended thinking, коли перша спроба агента не влучає в ціль — з більшим бюджетом на міркування він часто знаходить правильну відповідь.

---## 4. Патерни робочих процесів агентів

### Стандартний 5-кроковий цикл

Кожен основний агент (Copilot, Cursor, Claude Code) дотримується цього патерну:

```
1. DEFINE   — Ви описуєте завдання з контекстом
2. PLAN     — Агент аналізує кодову базу, пропонує підхід
3. PROPOSE  — Агент генерує diff-и / нові файли
4. CHECK    — Агент запускає тести, лінтер, перевірку типів
5. WRAP-UP  — Агент підсумовує зміни, ви робите огляд
```

**Ваша роль:** Кроки 1 (точне введення) і 5 (контроль якості). Агент виконує кроки 2–4.

### Відповідність автономії до розміру завдання

| Розмір завдання | Режим | Приклад | Ви контролюєте |
|-----------------|-------|---------|----------------|
| 1-5 рядків | **Inline** (автодоповнення) | Перейменування змінної, додавання перевірки на null | Прийняти/відхилити кожну пропозицію |
| 10-200 рядків | **Editor agent** (мультифайловий) | Додати API endpoint, написати набір тестів | Огляд плану + фінального diff |
| 200+ рядків | **Async agent** (фоновий) | Реалізувати повну функцію, рефакторинг модуля | Огляд специфікації + фінального PR |

### Коли НЕ варто використовувати агента

- **Невеликі швидкі правки**, де ваші пальці працюють швидше, ніж описувати зміну
- **Код із високим рівнем безпеки** (автентифікація, криптографія, платежі) — завжди людина як основний виконавець, агент як помічник
- **Вивчення нового API** — вам потрібно будувати ментальні моделі; делегування минає суть
- **Невизначене дослідження** — «зроби краще» дає сміття; агентам потрібні обмежені завдання

### Огляд коду за допомогою AI

Агенти корисні не лише для написання коду — вони чудово справляються з оглядом:

```
Prompt: "Review this PR diff. Check for:
1. Logic errors and missed edge cases
2. Security issues (injection, auth bypass, data exposure)
3. Performance concerns (N+1 queries, unnecessary allocations)
4. Deviations from our project conventions
5. Missing or inadequate test coverage"
```

**Вплив:** Виявляє проблеми, які людські рецензенти пропускають через втому або «замиленість ока». Особливо ефективно для:
- Великих PR (200+ рядків), де людська увага знижується
- Змін, пов'язаних із безпекою, де додатковий погляд критично важливий
- Міжкомандних PR, де рецензенти не мають контексту предметної області

**Найкраща практика:** Використовуйте AI-огляд як перший прохід, а не заміну. Агент позначає потенційні проблеми → людський рецензент фокусується на архітектурних та бізнес-логічних рішеннях.

### Генерація документації

Найпростіше завдання для агента з найвищою окупністю — і найкращий спосіб переконати скептиків:

| Тип документації | Патерн промпту | Зекономлений час |
|------------------|----------------|------------------|
| API-документація з коду | "Generate OpenAPI spec from these route handlers" | 2-4 години → 5 хвилин |
| Architecture Decision Records | "Document the architectural decisions visible in this module: patterns used, tradeoffs made, alternatives rejected" | 1-2 години → 10 хвилин |
| Оновлення README | "Update README to reflect current project structure, setup steps, and available scripts" | 30-60 хв → 3 хвилини |
| Посібники з онбордингу | "Generate an onboarding guide for a new developer joining this project, based on the codebase structure and conventions" | 4-8 годин → 15 хвилин |
| Changelog із комітів | "Generate a changelog for the last 20 commits, grouped by feature/fix/chore" | 20-30 хв → 1 хвилина |

**Чому це переконує скептиків:** Кожна команда має борг документації. Коли агент генерує точну документацію за хвилини замість годин ручної роботи, цінність стає безперечною.

### Реальний приклад: реалізація REST endpoint

```
Task: Add GET /api/v2/users/:id/activity endpoint

Step 1 (you): Write EARS spec
- When a valid user ID is provided, the system shall return the last 
  50 activity events sorted by timestamp descending
- If the user ID does not exist, the system shall return 404
- Where the caller lacks `read:activity` scope, the system shall return 403

Step 2 (agent): Reads spec, finds existing endpoint patterns in /api/v2/,
  identifies UserActivity model, proposes file structure

Step 3 (agent): Generates controller, service, DTO, and test files 
  following existing patterns

Step 4 (agent): Runs `npm test`, fixes one import path, re-runs → green

Step 5 (you): Review diff — 4 files, ~180 lines. Approve after 
  adjusting one query optimization. Total time: 12 minutes.
```

Традиційна оцінка для цього ж завдання: 45–90 хвилин.

## 5. MCP Integration — Підключення агентів до вашого стеку

### Що таке MCP

**Model Context Protocol** — це відкритий стандарт, який дозволяє AI-агентам підключатися до зовнішніх систем — баз даних, API, інструментів управління проєктами — через уніфікований інтерфейс. Уявіть собі USB-C для AI-інструментів: один протокол, багато інтеграцій.

### Практичні інтеграції

| Інтеграція | Що робить | Вплив |
|------------|-----------|-------|
| **Jira / Linear** | Агент читає деталі тікетів, переходить за пов'язаними специфікаціями, оновлює статус після завершення | Більше не потрібно копіювати описи тікетів у промпти |
| **GitHub** | Читає OpenAPI / Protobuf контракти, виявляє зламні зміни, створює PR | Contract-first розробка стає контрольованою |
| **Figma** | Витягує дизайн-специфікації, генерує скелети компонентів, чеклісти макетів | Дизайнери та агенти говорять однією мовою |
| **Database** | Створює seed-дані, перевіряє поточний стан схеми, генерує скрипти міграцій | Агент працює з реальними формами даних, а не з припущеннями |
| **Browser** | Навігує сторінками, перевіряє рендеринг UI, аналізує DOM / консоль / мережу | Валідація фронтенду без ручних QA-циклів |

**Час налаштування:** Зазвичай 1–2 хвилини на інтеграцію. Більшість агентів автоматично пропонують конфігурацію.

### Модель безпеки MCP

Це обов'язкова вимога для корпоративного впровадження:

- **Мінімальні привілеї** — Кожен MCP-сервер отримує лише ті дозволи, які йому потрібні
- **Явні підтвердження** — Деструктивні операції (DELETE, DROP, force-push) вимагають підтвердження людини
- **OAuth/OIDC** — Стандартна автентифікація; жодних API-ключів у відкритому тексті
- **PII-фільтрація** — Видалення конфіденційних даних перед потраплянням у контекст моделі
- **Захист від prompt injection** — Валідація всіх відповідей MCP перед обробкою
- **Аудит-журнали** — Логування кожного виклику інструменту для перевірки відповідності
- **Пісочниця для запису** — Зміни файлів проходять через огляд, ніколи напряму в продакшн

## 6. Testing & Validation — AI робить TDD практичним

### Проблема вартості TDD — вирішена

Головне заперечення проти TDD завжди було: *«Написання тестів займає надто багато часу.»* AI усуває це заперечення. Коли ви можете згенерувати комплексний набір тестів за секунди зі специфікації, окупність TDD різко зростає на вашу користь.

**Вплив: 100% покриття тестами стає стандартом**, оскільки основний біль — написання тестів — усунено.

### Патерн специфікації для кожного тесту

Замість `"write tests for this module"` специфікуйте кожен тест:

```markdown
### Test: User creation with valid data
- **Preconditions:** Empty database, valid UserCreateDTO
- **Steps:** Call createUser(dto)
- **Expected:** Returns User with generated UUID, createdAt set to now ±1s
- **Mocks:** UserRepository.save() returns the entity with generated ID
```

Це дає точні, рецензовані тести. Агент не вгадує — він виконує вашу специфікацію.

### Патерн нарощування

```
1. Happy path         — Чи працює базовий потік?
2. Negative cases     — Невалідні дані, відсутні дані, неавторизований доступ
3. Edge cases         — Конкурентність, граничні значення, порожні колекції
4. Integration tests  — HTTP + база даних разом для повної валідації сценарію
```

**Крива вартості:** Перший тест у модулі займає найбільше часу (агент вивчає патерни). Кожен наступний тест стає дешевшим, оскільки конвенції модуля стабілізуються.

### Самовалідація агентом

Надайте агентам інструменти для перевірки їхнього власного результату:

| Домен | Інструмент валідації | Що перевіряє |
|-------|---------------------|--------------|
| Backend | Test runner (Jest, Vitest, pytest) | Коректність логіки, відповідність контракту |
| Frontend | Порівняння скриншотів | Візуальна регресія, точність макету |
| Mobile | ADB / Simulator tools | Коректність рендерингу, потік взаємодії |
| API | HTTP client + assertions | Коди статусу, форми відповідей, заголовки |

Коли агент може запускати тести та виправляти помилки автономно, ви отримуєте коротший цикл зворотного зв'язку, ніж при TDD, керованому людиною.

### Внутрішні конвеєри оцінки (Evaluation-Driven Development)

Окрім тестування окремих функцій, високоефективні команди створюють **«Evaluation-Driven Development» (EDD)** — внутрішній, ізольований у Docker стенд, де агенти тестуються на історичних багах та запитах на функціональність компанії.

**Чотири виміри для вимірювання:**

| Вимір | Що вимірювати | Як |
|-------|---------------|-----|
| **Correctness** | Відсоток проходження unit- та integration-тестів згенерованих патчів | Автоматизований test runner у CI |
| **Security** | Результати сканування вразливостей AI-згенерованого коду | Sonar, Snyk у пайплайні |
| **Maintainability** | Читабельність, дотримання стилю, якість документації | LLM-as-judge оцінювання |
| **Token Efficiency** | Вартість і кількість кроків до розв'язання | Відстеження tokens/resolved-issue |

**Чому публічних бенчмарків недостатньо:** SWE-bench Verified був визнаний застарілим через підтверджене забруднення даних — провідні моделі запам'ятовували рішення з навчальних даних, завищуючи оцінки до ~70%+. Його замінник, **SWE-bench Pro** (курований OpenAI, запущений у 2025), використовує відкладені завдання: провідні моделі набирають 43-46%, досягаючи ~57% із просунутим scaffolding. Навіть продуктивність SWE-bench Pro на приватних zero-shot репозиторіях значно падає. Ваша внутрішня оцінка на вашій кодовій базі — єдина метрика, що має значення для вашої команди.## 7. Організація коду для зрозумілості AI

### DDD + AI: природні союзники

Принципи Domain-Driven Design ідеально відповідають AI-native розробці:

| Принцип DDD | Перевага для AI |
|---------------|------------|
| Bounded contexts | Контекст агента залишається малим і сфокусованим |
| Ubiquitous language | Послідовне іменування = менше галюцинацій |
| Чіткі контракти | Агент точно знає, які вхідні/вихідні дані створювати |
| Межі агрегатів | Природна декомпозиція на атомарні завдання |

**AI знижує вартість впровадження DDD.** Традиційний аргумент проти DDD — «забагато шаблонного коду». Коли генерація шаблонного коду безкоштовна, DDD стає очевидним вибором.

### Що допомагає агентам

- **Однозначні імена** — `calculateOrderTotal()`, а не `calc()` чи `process()`
- **Явне замість магічного** — Прямі імпорти, жодної прихованої DI-магії, прозорий потік даних
- **Чіткі межі модулів** — Кожен модуль має зрозумілий публічний API; внутрішня реалізація — приватна
- **Файли поруч** — Тести поруч із вихідним кодом, типи поруч із реалізацією
- **README для кожного модуля** — Короткий опис призначення, ключових патернів та точок входу

### Що ламає агентів

| Антипатерн | Чому не працює | Рішення |
|-------------|-------------|-----|
| Заплутані імпорти | Агент не може відстежити потік даних | Забезпечити межі модулів |
| Незрозуміле іменування | `handleStuff()` → агент вгадує неправильно | Перейменувати відповідно до наміру: `handlePaymentWebhook()` |
| Перевантажені базові класи | God-класи на 1000 рядків переповнюють контекст | Декомпозувати на сфокусовані сервіси |
| Динамічне метапрограмування | Невидима поведінка плутає планування | Надавати перевагу явним патернам |
| Мертвий код | Засмічує контекст, вводить агента в оману | Видалити — AI робить повторну генерацію тривіальною |

## 8. Практики безпеки та захисту

### Непорушні правила

1. **Лише корпоративні ліцензії.** Ніколи не використовуйте особисті AI-акаунти для корпоративного коду. Управління даними вимагає корпоративних планів із контролем зберігання даних і журналюванням аудиту.

2. **Людський огляд для всіх критичних з точки зору безпеки шляхів.** Автентифікація, авторизація, обробка платежів, криптографія — AI допомагає, людина вирішує.

3. **Ставтеся до результатів агента як до роботи молодшого розробника.** Переглядайте все. Не довіряйте нічому неявно. Модель впевнена навіть коли помиляється.

### Реальна оцінка автономності агентів (весна 2026)

Обіцянка: *«Повністю автономні агенти, що створюють повноцінні функції без нагляду».*

Реальність: **Практична автономність = чітко специфіковані, жорстко обмежені атомарні завдання.** Повна автономність у складних мультисистемних функціях все ще ближча до фантазії, ніж до продакшену. Агенти, які працюють, — це ті, яким надано:
- Чіткі специфікації (EARS / SDD)
- Обмежений обсяг (вміщується в контексті)
- Інструменти самоперевірки (тести, лінтери)
- Людські шлюзи в критичних точках прийняття рішень

### Ліцензійний ризик

LLM навчені на відкритому коді. Вони можуть генерувати фрагменти, що нагадують код під ліцензіями GPL, AGPL або іншими copyleft-ліцензіями. **Заходи зменшення ризику:**
- Запускайте сканери ліцензій (FOSSA, Snyk) для всього згенерованого коду
- Ставтеся до AI-згенерованого коду з такою ж ретельністю дотримання вимог, як і до сторонніх залежностей
- Документуйте, який код був згенерований AI, для аудиторських перевірок

### Захист від prompt injection

Коли агенти взаємодіють із зовнішніми даними (введення користувача, відповіді API, вміст бази даних), зловмисний контент може перехопити поведінку агента. **Заходи зменшення ризику:**
- Валідуйте та очищуйте всі відповіді MCP-серверів
- Ніколи не передавайте необроблений контент користувача безпосередньо в системні промпти
- Використовуйте парсинг вихідних даних зі строгими схемами
- Моніторте дії агента на предмет неочікуваних патернів

### Регуляторна відповідність (весна 2026)

Правове середовище прискорилось. Інженерні команди тепер мають враховувати:

**EU AI Act (повністю набирає чинності 2 серпня 2026):**
- **Стаття 50**: Зобов'язання щодо прозорості вимагають розкриття, коли контент згенерований AI. Однак Стаття 50 **явно звільняє** AI-допоміжний контент, що проходить людський огляд перед публікацією. Внутрішній вихідний код, що проходить через процеси PR-рев'ю з людським затвердженням, підпадає під це звільнення. Командам все одно варто підтримувати VCS-атрибуцію AI-згенерованого коду як найкращу практику для відстежуваності.
- **Класифікація високого ризику**: Якщо AI-згенерований код є компонентом безпеки в регульованих продуктах (медичні пристрої, критична інфраструктура), застосовуються Статті 8-15 — що вимагають ретельного управління ризиками, належного управління даними та задокументованого людського нагляду.
- **Заборонені практики**: Системи, що займаються шкідливою маніпуляцією або соціальним скорингом, суворо заборонені.

**Інтелектуальна власність та авторське право:**
Бюро з авторських прав США (USCO) постановило, що AI-згенерований контент без значного людського творчого внеску **не підлягає захисту авторським правом**. Критично важливо, що USCO зазначило: робочий процес «людська специфікація → виконання AI → людський огляд» сам по собі **недостатній** для встановлення авторського права — людина повинна внести суттєві творчі зміни до результатів AI, а не просто переглянути та затвердити їх. Модель SDD забезпечує найнадійнішу доступну основу, але інженери повинні активно модифікувати та формувати згенерований код (а не просто приймати його), щоб встановити «творчий вираз», необхідний для захисту авторським правом. Країни-члени ЄС розробляють подібні (хоча ще не уніфіковані) позиції. Документуйте ланцюжок від специфікації до коду та модифікацій.

**Практичний чекліст відповідності:**

| Вимога | Реалізація | Стандарт |
|------------|---------------|----------|
| **Відстежуваність** | Незмінний аудиторський слід кожного промпту та відповіді, що згенерували продакшн-код | SOX, PCI-DSS |
| **Очищення PII** | Ізольований за тенантами інференс-шлюз видаляє персональні дані перед викликами хмарних LLM | GDPR |
| **Відповідальність за вразливості** | Сканування безпеки на швидкості машини (Sonar, Snyk) для всього AI-згенерованого коду | Усі |
| **Відстеження ліцензій** | Шари спостережуваності для виявлення copyleft-ліцензованого коду (GPL, AGPL) у результатах AI | IP compliance |
| **Маркування коду** | Машиночитані маркери, що ідентифікують AI-згенерований код у метаданих VCS | EU AI Act (найкраща практика; зверніть увагу на звільнення за Ст. 50 для коду з людським оглядом) |
| **Людський нагляд** | Задокументовані шлюзи людського огляду для всього продакшн-коду | EU AI Act Ст. 14 |

Організації, що будують залежність від ціни $20/місяць, повинні планувати $100-300/місяць, коли ціни нормалізуються. Враховуйте це в розрахунках TCO вже зараз.

| Середній рівень | Llama 4 (Scout/Maverick) / Mistral Large | -10% до -20% | RAG-орієнтований пошук знань |
| Flash-клас | Codestral / Phi-4 | -20% до -30% | Автодоповнення з низькою затримкою |

Провідні open-source моделі скорочують розрив з пропрієтарними frontier-моделями на бенчмарках кодування, деякі з них досягають результатів, порівнянних з ранніми версіями GPT-4. Компанії успішно дотренировують open-source моделі (використовуючи Unsloth або LoRA) на внутрішніх кодових базах, дозволяючи моделі з 70B параметрів перевершувати більшу загальну модель на доменно-специфічних завданнях.

**Гібридні патерни приватності** для регульованих галузей:
1. **PII Stripping Proxy**: Вихідні виклики LLM проходять через локальний шлюз (Bifrost, LiteLLM), що токенізує чутливі дані
2. **Маршрутизація за класифікацією даних**: Прості завдання → локальні моделі (Mistral 7B); складне міркування → захищений корпоративний хмарний інстанс
3. **Локальний RAG + хмарне міркування**: База знань (Qdrant) залишається on-premise; лише анонімізовані запити на міркування йдуть у хмару

Вартість обладнання для потужного on-premise GPU-кластера (наприклад, 8x NVIDIA B200) починається від ~$360 000-400 000 лише за GPU ($45-50 тис. за B200), з загальними витратами на розгортання (мережа, сховище, охолодження, стійкова інфраструктура), що сягають приблизно **$515 000** за 8-GPU сервер (еталонна конфігурація Lenovo ThinkSystem SR685a V4, 2026). TCO при масштабі становить приблизно **$21,40/годину** з повним навантаженням. On-premise демонструє **8-кратну перевагу у вартості** порівняно з хмарним IaaS та **18-кратну порівняно з MaaS API** при стабільному навантаженні, з точкою окупності **менше ніж за 4 місяці** (аналіз TCO від Lenovo, 2026).

## 11. Робота з Legacy-кодом — найвпливовіший варіант використання

### Чому модернізація legacy-коду — це кілер-застосування

Якщо ви шукаєте єдине застосування AI-native практик із найвищою окупністю інвестицій — це legacy-код. У кожної організації є модулі, до яких ніхто не хоче торкатися: без документації, без тестів, написані людьми, що пішли з компанії роки тому. AI-агенти перетворюють це з нелюбимої рутини на системний процес із низьким ризиком.

### П'ятикроковий паттерн для legacy-коду

```
1. READ       — Агент завантажує legacy-модуль (до повного контекстного вікна)
2. UNDERSTAND — Агент пояснює, що робить код, картографує залежності, визначає паттерни
3. DOCUMENT   — Агент генерує вбудовану документацію, README модуля, архітектурні нотатки
4. TEST       — Агент пише тести на поточну поведінку (golden master / характеризаційні тести)
5. REFACTOR   — Маючи тести як страхувальну сітку, агент поступово модернізує код
```

**Критичний порядок:** Завжди генеруйте тести ДО рефакторингу. Тести фіксують поточну поведінку — включно з багами, які ви, можливо, захочете зберегти для зворотної сумісності. Це страхувальна сітка, що робить рефакторинг безпечним.

### До та після: реальний legacy-модуль

**До:** 2 000 рядків `OrderProcessor.java` — без тестів, змішані відповідальності, 15 років латок.

```
Prompt: "Analyze this file. Explain its responsibilities, identify the distinct 
concerns it handles, and generate characterization tests that capture its 
current behavior. Then propose a refactoring plan that separates concerns 
into focused classes."
```

**Після (30 хвилин роботи агента + перевірка людиною):**
- 47 характеризаційних тестів, що покривають усі гілки → страхувальну сітку встановлено
- README модуля з документацією бізнес-правил, граничних випадків та точок інтеграції
- План рефакторингу: 5 виділених класів (OrderValidator, PricingEngine, InventoryReserver, PaymentProcessor, NotificationDispatcher)
- Кожне виділення — це окремий PR для перегляду з тестами, що проходять до і після

**Традиційна оцінка:** 2–3 тижні обережної ручної роботи. **AI-native:** 2–3 дні, включно з ретельною перевіркою людиною.

### Паттерни міграцій, з якими агенти справляються добре

| Тип міграції | Підхід агента | Роль людини |
|---------------|---------------|------------|
| Оновлення фреймворку (напр., Angular 14→17) | Агент читає гід з міграції + кодову базу, застосовує зміни систематично | Перевірка ламаючих змін, тестування інтеграції |
| Міграція мови (напр., JS→TS) | Агент додає типи поступово, модуль за модулем | Перевірка точності типів, робота зі складними дженериками |
| Оновлення версії API | Агент читає нову документацію API, оновлює всі точки виклику | Верифікація бізнес-логіки, тестування граничних випадків |
| Міграція бази даних | Агент генерує міграційні скрипти з різниці схем | Перевірка цілісності даних, тестування відкату |

### Метрики впливу

Завдання з рефакторингу legacy-коду, які займали **тижні**, тепер займають **дні**. Найбільша економія досягається на документуванні та генерації тестів — тих частинах, які розробники пропускають при ручній роботі з legacy-кодом.

## 12. Дорожня карта впровадження в команді — від першого використання до масштабу організації

### Поетапний план розгортання

#### Тиждень 1: Індивідуальне дослідження
- Кожен розробник обирає свій бажаний інструмент (див. Розділ 1)
- Спробуйте три швидкі перемоги: генерація тестів, генерація документації, рефакторинг із поясненням
- Жодних змін процесів поки що — просто набувайте досвіду
- **Метрика успіху:** Кожен член команди виконав хоча б одне продуктивне завдання з AI-агентом

#### Місяць 1: Командні конвенції
- Створіть файли конвенцій проєкту (`.github/copilot-instructions.md`, `AGENTS.md`)
- Напишіть першу SDD-специфікацію для реального завдання спринту
- Встановіть практики перегляду: як переглядати AI-згенерований код, на що звертати увагу
- Почніть документувати, що працює, а що ні, у спільній вікі
- **Метрика успіху:** Файл конвенцій закомічено, перша SDD-специфікація використана у спринті

#### Квартал 1: Інтеграція у процеси
- Налаштуйте MCP-інтеграції для ваших основних інструментів (GitHub, Jira, база даних)
- SDD стає процесом за замовчуванням для середніх і великих завдань
- Починає працювати маховик compound engineering: специфікації стають шаблонами, конвенції вдосконалюються
- Починайте відстежувати метрики швидкості та покриття
- **Метрика успіху:** MCP налаштовано, >50% фіч побудовано за SDD, вимірюване покращення швидкості

#### Квартал 2+: Масштаб організації
- Спільна бібліотека специфікацій між командами
- Міжкомандні конвенції (контракти API, стандарти іменування, архітектурні паттерни)
- Формальна панель відстеження метрик: швидкість, покриття, рівень дефектів, задоволеність розробників
- Бібліотека Skills: командні Skills для агентів, які спільно використовуються між проєктами
- **Метрика успіху:** Впровадження на рівні організації, задокументоване зростання продуктивності, зростаюча бібліотека Skills

#### Квартал 3+: Інституційна платформа (100+ інженерів)
- **Спільні репозиторії специфікацій**: Централізовані Git-репозиторії з EARS-шаблонами, «Golden Prompts» та архітектурними конвенціями для всіх команд
- **Централізовані MCP-шлюзи**: Уніфікований каталог інтеграцій інструментів (Slack, Jira, корпоративні БД) під управлінням RBAC — замінює налаштування MCP окремо для кожної команди
- **Внутрішня AI-платформа**: Портал для розробників (Backstage, Spotify Portal) як центр AI-ready інженерної культури — спільні специфікації, бібліотеки конвенцій, реєстр Skills для агентів
- **Спільноти практик**: Міжкомандний обмін знаннями — поширення досвіду від піонерів до послідовників через регулярні демо, спільні playbook-и та внутрішні воркшопи
- **Метрика успіху:** Впровадження платформи >80%, міжкомандне повторне використання специфікацій, швидкість онбордингу скорочується з тижнів до годин

### Як SDD вписується в наявні процеси

| Наявна практика | Як вона змінюється |

| Метрика | Що вимірює | Чому це важливо |
|--------|-----------------|----------------|

| **Темп скорочення технічного боргу** | Модернізованих legacy-модулів / квартал | Відстежує організаційне здоров'я |
| **AI-Attributed Change Failure Rate** | Виробничі інциденти від AI-згенерованого коду | Сигнал безпеки щодо якості агента |
| **Spec Reuse Rate** | % завдань, що використовують шаблонні специфікації | Вимірює зрілість compound engineering |

---## 13. Compound Engineering — Ефект маховика

### 5-й стовп

Чотири стовпи AI-native розробки — SDD, контекстний інжиніринг, агентні робочі процеси, інтеграція MCP — будуються один на одному. Але є 5-й, який змушує всю систему прискорюватися: **compound engineering**.

Кожна специфікація, файл конвенцій, інтеграція MCP та визначення Skills накопичуються як **організаційний капітал знань**. На відміну від коду (який знецінюється), цей капітал зростає з часом.

### Як працює маховик

```
Специфікації стають шаблонами  →  Шаблони прискорюють створення нових специфікацій
    ↑                                              ↓
Конвенції вдосконалюються      ←  Результат агента покращується  ←  Кращий контекст
    ↑                                              ↓
Skills накопичуються           →  Нові проєкти стартують швидше
    ↑                                              ↓
Нові учасники адаптуються швидше  ←  Кодова база самодокументується
```

### Конкретний приклад: Місяць 1 проти Місяця 6

**Місяць 1 — Створення CRUD API:**
- Написання специфікації з нуля (~30 хв)
- Налаштування файлу конвенцій (~15 хв)
- Агент генерує код, потребує значного рев'ю (~20 хв на перевірку)
- Написання тестів з агентом (~15 хв)
- **Загалом: ~80 хвилин**

**Місяць 6 — Створення аналогічного CRUD API:**
- Клонування шаблону специфікації, адаптація під конкретну сутність (~5 хв)
- Конвенції вже існують і були вдосконалені через 50+ завдань
- Агент генерує код, що точно відповідає патернам команди (~5 хв на перевірку)
- Агент генерує тести, що відповідають патернам команди (~3 хв на перевірку)
- **Загалом: ~15 хвилин**

**Те саме завдання, у 5 разів швидше.** І якість вища, бо конвенції пройшли бойове тестування.

### Ефекти накопичення знань

| Що накопичується | Як | Вплив на масштабі |
|---------------|-----|----------------|
| **Специфікації** | Попередні специфікації стають шаблонами; з'являються патерни | Нові фічі специфікуються за хвилини, а не години |
| **Файли конвенцій** | Вдосконалюються з кожним проєктом; граничні випадки документуються | Якість результатів агента постійно зростає |
| **Бібліотека Skills** | Командні Skills спільно використовуються між проєктами | Нові проєкти успадковують AI-playbook команди |
| **Інтеграції MCP** | Налаштовуються один раз, використовуються всюди | Нові проєкти підключені до інструментів з першого дня |
| **Патерни рев'ю** | Команда вчиться, що перевіряти в AI-результатах | Час на перевірку зменшується, якість зростає |

### Конкурентний рів

Організації, які починають накопичувати зараз, будують **перевагу, що прискорюється**. До моменту, коли конкуренти почнуть впроваджувати AI-native практики, ранні послідовники матимуть:
- Відточені шаблони специфікацій, що покривають усю їхню доменну область
- Бойові конвенції, що забезпечують надійний результат агента
- Бібліотеки Skills, що кодують роки накопичених знань про робочі процеси
- Команди, вільно володіючі написанням специфікацій, контекстним інжинірингом та AI code review

Цей капітал знань важко скопіювати, і він зростає з використанням.

|--------|------------|---------------------------|

### Розрахунок окупності

```
Вартість інструментів: $200/місце/місяць × 12 = $2,400/рік
Вартість розробника:   $150,000/рік ÷ 2,000 годин = $75/година
Зекономлений час:      730 годин × $75 = $54,750/рік
Чиста вигода:          $54,750 - $2,400 = $52,350/рік на розробника
ROI:                   ~2,100%
Термін окупності:      < 1 місяць
```

Навіть при 50% від оцінених заощаджень часу ROI залишається впевнено позитивним.

### Багаторічний TCO-фреймворк

Для організацій, що планують на перспективу, використовуйте галузево-стандартну формулу TCO:

$$TCO = C_{acq} + (C_{op} + C_{maint} + C_{supp}) \times Y + C_{token} - V_{res}$$

Де $Y$ — очікуваний термін використання (зазвичай 5 років), а $V_{res}$ — залишкова вартість.

**5-річне порівняння для організації з 200 інженерами:**

| Категорія витрат | Cloud-Native (API/MaaS) | On-Premise (GPU Cluster) |
|--------------|-------------------------|---------------------------|
| Початкові капітальні витрати | $0 | ~$515,000 (8x B200 сервер, повністю сконфігурований) |
| Річне обслуговування/електроенергія | $0 | ~$100,000-130,000 |
| Середня вартість токенів (річна) | Залежить від використання та провайдера | $0 (власний інференс) |
| **5-річна загальна вартість** | **Залежить від використання** | **~$1,015,000-1,165,000** |
| Цінова перевага | Гнучкість, відсутність початкових витрат | Потенційно значна економія на токен при високому завантаженні |

Cloud пропонує гнучкість для доступу до передових моделей міркування, які ще не доступні локально. On-premise забезпечує передбачувані витрати та суверенітет даних. Більшість підприємств обирають **гібридний підхід**: локальні моделі для високообʼємних/низькоскладних задач + хмара для складного міркування. Фактична цінова перевага on-premise суттєво залежить від рівня завантаження, витрат на персонал та інфраструктурних накладних — організаціям слід моделювати свої конкретні патерни використання перед прийняттям рішення.

### Приховані витрати, які слід закласти в бюджет

| Прихована вартість | Опис | Пом'якшення |
|------------|-------------|------------|
| **Просідання продуктивності** | 3-6 місяців калібрування (Gartner, 2025) | Плануйте це; не очікуйте миттєвих результатів |
| **Контекстний інжиніринг** | Час на підготовку Repository Context та метаданих | Створюйте файли конвенцій заздалегідь; накопичуйте з часом |
| **AI-борг** | Код, який ніхто не розуміє, бо його написав AI | Вимагайте трасування від специфікації до коду; обовʼязкове рев'ю |
| **Дебагінг AI-коду** | Інші патерни збоїв, ніж у коді, написаному людиною | Навчайте "форензик-дебагінгу" — використовуйте reasoning traces |
| **Зміна профілю найму** | Потрібні "гібридні професіонали", а не "чисті кодери" | Перенавчайте існуючу команду; оновлюйте описи вакансій |

### Формулювання, які працюють з керівництвом

**Не кажіть:** "Замінити розробників на AI."
**Кажіть:** "Доставляти більше тією ж командою."

| Занепокоєння керівництва | Ваша відповідь |
|-------------------|---------------|
| Ефективність штату | "Та сама команда випускає у 2-3 рази більше фіч за квартал" |
| Час виходу на ринок | "Від запиту на фічу до продакшену — дні замість тижнів для специфікованих завдань" |
| Якість | "Покриття тестами зростає з 60-80% до 90%+, бо вартість TDD усунено" |
| Зниження ризиків | "Менша залежність від трайбальних знань — кодові бази стають самодокументованими" |
| Утримання талантів | "Інженери працюють над дизайном та архітектурою, а не шаблонним кодом — вища задоволеність роботою" |

### Реальність ціноутворення

Будьте прозорими щодо витрат:
- Поточні споживчі ціни ($10-50/місяць), як вважається, нижчі за довгостроково стійкий рівень, оскільки вендори захоплюють частку ринку
- Плануйте $100-300/місяць на місце, коли ціни нормалізуються
- Навіть за повною ціною ROI залишається впевнено позитивним (див. розрахунок окупності вище)
- Пряме API-ціноутворення (~$1,000/місяць при інтенсивному використанні) дає уявлення про несубсидований рівень витрат

### Вартість НЕвпровадження

Це аргумент, який закриває угоду:
- **Прискорення конкурентів:** Організації, що впровадили AI-native практики, доставляють у 2-3 рази швидше. Ваші конкуренти вже це роблять.
- **Ризик втрати талантів:** Топові інженери дедалі більше очікують AI-native інструментарій. Його відсутність стає недоліком для утримання та рекрутингу.
- **Розрив у накопиченні знань:** Кожен місяць затримки — це місяць, протягом якого конкуренти будують свої бібліотеки специфікацій, конвенцій та Skills — капітал, який не можна купити пізніше.

## 15. Поширені заперечення та контраргументи

### "AI пише поганий код"

**Контраргумент:** Якість є функцією якості вхідних даних. Розпливчасті промпти дають розпливчастий код — це проблема специфікації, а не AI. З SDD (точні специфікації + EARS-синтаксис + ворота рев'ю) AI стабільно створює код, який проходить перевірку з першого разу для добре специфікованих завдань.

**Покажіть, а не розповідайте:** Проведіть демо з порівнянням. Те саме завдання: один раз з розпливчастим промптом ("побудуй сервіс нотифікацій"), один раз зі SDD-специфікацією. Різниця драматична і помітна за хвилини.

### "Ми втратимо інженерні навички"

**Контраргумент:** Інженери піднімаються ВГОРУ по стеку, а не виходять зі стеку. Навички, які стають важливішими, ніж будь-коли:
- Проєктування систем та архітектура (цінніше за імплементацію)
- Написання специфікацій (нова ключова навичка)
- Code review та оцінка якості (складніше, а не простіше)
- Контекстний інжиніринг та дизайн промптів (абсолютно нова навичка)

Аналогія: Компілятори не зробили програмістів непотрібними — вони змістили роботу з асемблера на проєктування вищого рівня. AI робить те саме для наступного рівня абстракції.

### "Це просто хайп / чергова мода"

**Контраргумент:** Покажіть цифри з Розділу 9. У 2-5 разів швидше генерація коду для специфікованих завдань, усунення вартості TDD, документація за хвилини. Це вимірювані результати ранніх послідовників, а не прогнози. Технологія готова до продакшену вже сьогодні — використовується інженерними командами у масштабі по всій індустрії.

Також: кожен великий вендор IDE (Microsoft, JetBrains), кожен великий хмарний провайдер (AWS, Google, Azure) та кожна провідна AI-лабораторія активно інвестують. Це не побічний хобі-проєкт — це основний напрямок розробницького інструментарію.

### "Ризик безпеки"

**Контраргумент:** Обґрунтоване занепокоєння — і вже вирішене корпоративними контролями:
- Корпоративні плани з контролем збереження даних та журналами аудиту
- Ворота рев'ю: весь AI-згенерований код проходить стандартний процес перевірки
- Модель безпеки MCP: OAuth/OIDC, принцип найменших привілеїв, фільтрація PII, журнали аудиту
- Сканування ліцензій усього згенерованого коду (FOSSA, Snyk)
- Ваш код не використовується для навчання (корпоративні угоди)

Модель безпеки для AI-згенерованого коду суворіша, ніж та, яку більшість команд застосовує до коду, написаного людиною.

### "Наша кодова база надто складна / надто специфічна"

**Контраргумент:** Модернізація застарілих систем — це насправді сценарій з *найбільшим впливом* (Розділ 11). Складні, недокументовані кодові бази отримують найбільше від AI-агентів, які можуть читати цілі модулі, генерувати документацію та писати характеризаційні тести. Чим складніша ваша кодова база, тим більше часу заощаджує AI.

### "Це надто дорого"

**Контраргумент:** Див. Розділ 14. Навіть за повною несубсидованою ціною ($200/місце/місяць) термін окупності — менше місяця. Питання не в тому, чи можете ви дозволити собі впровадження — а в тому, чи можете дозволити собі не впроваджувати, поки конкуренти прискорюються.

### "А як щодо регуляторної відповідності?"

**Контраргумент:** EU AI Act (серпень 2026) створює нові вимоги до прозорості та нагляду, але Стаття 50 **явно звільняє** контент, створений за допомогою AI, що проходить людську перевірку — це означає, що внутрішній код, який проходить PR-процеси з людським затвердженням, звільнений від зобов'язань щодо водяних знаків/розкриття. Організації, що практикують SDD з трасуванням, воротами людського рев'ю та журналами аудиту, добре підготовлені до відповідності. Щодо авторського права, зверніть увагу, що USCO вимагає від інженерів внесення суттєвих творчих модифікацій до AI-результатів (а не лише перегляд/затвердження) — SDD забезпечує фреймворк, але активне людське формування коду є необхідним. Див. Розділ 8 для повного чеклісту відповідності.## 16. Agentic CI/CD — Патерни інтеграції

### Від статичних перевірок до безперервної якості

Станом на 2026 рік приблизно **40% корпоративних** інженерних організацій інтегрують спеціалізованих AI-агентів у пайплайни доставки (Gartner, прогноз 2025 року щодо agentic AI у розробці ПЗ до 2028). Агенти — це не просто «перевірки» — вони активні учасники, які діагностують збої, оновлюють специфікації та самостійно виправляють зламані тести.

### Патерни Agentic CI/CD

| Фаза CI/CD | Агентний патерн | Практичний інструментарій |
|-------------|----------------|-------------------|
| Pull Request | Автоматичне резюмування PR та логічний ревʼю | GitHub Actions + Claude Code |
| Валідація специфікації | Гейт відповідності EARS-синтаксису | Кастомні правила лінтингу або валідатори специфікацій |
| Помилки збірки | Автономний аналіз кореневих причин (RCA) | Instana + AI Agent-ремедіатори |
| Генерація тестів | Природна мова → Playwright/Cypress | mabl AI Agent |
| Оркестрація IaC | Агентне виправлення дрифту Terraform/Pulumi | Firefly AI |

### Стратегія Shift-Left Intelligence

1. **Тригери пайплайну**: Кожен коміт запускає «Діагностичного агента», який аналізує diff на відповідність архітектурі з репозиторієм специфікацій — а не лише синтаксичні помилки.
2. **Адаптивні гейти якості**: Агенти оцінюють ризик деплойменту порційно. Візуальна регресія на сторінці з низьким трафіком позначається для ревʼю; збій у логіці checkout-потоку викликає «Автоматичне блокування деплойменту».
3. **Автономне створення тестів**: Інженери описують бажану валідацію природною мовою. Агент генерує e2e-тест та інтегрує його в CI-набір.
4. **Гейт відповідності EARS**: Нові PR валідуються відповідно до EARS-специфікацій проєкту. Вимоги без відповідних тестів позначаються.

### Вартість AI у CI

Інтенсивне використання AI обходиться у $100–200/розробник/місяць у CI-пайплайнах. Для команди з 200 осіб: $20 000–$40 000/місяць. **Мітигація:**
- **AI Gateways** для моніторингу витрат на токени та ієрархій бюджетів
- **Рівневе використання агентів**: Простий лінтинг = локальна модель; складний ревʼю = frontier-модель
- **Кешування**: Кешування аналізу агентів для ідентичних патернів коду

## 17. Моніторинг та спостережуваність після деплойменту

### Недетермінована реальність

AI-згенерований код демонструє інші патерни збоїв, ніж код, написаний людиною. Традиційний APM (Application Performance Monitoring) недостатній — командам потрібна **спостережуваність, яка розуміє поведінку коду, а не лише аптайм**, щоб виявляти поведінковий дрифт та помилки в міркуваннях.

### Специфічні для AI патерни збоїв

| Патерн | Опис | Виявлення |
|---------|-------------|-----------|
| **Correctness Decay** | Логіка синтаксично правильна, але семантично хибна для бізнес-кейсу | Моніторинг property-based тестів у продакшені |
| **Prompt Regression** | Якість виводу агента погіршується при зміні контексту специфікації | Порівняння виводу до/після |
| **Edge Case Blindness** | AI обробляє happy path, але пропускає доменно-специфічні граничні випадки | Канаркове розгортання з порівнянням з еталоном |
| **Pattern Overfitting** | AI відтворює поширені патерни, навіть коли задача вимагає іншого підходу | Архітектурний ревʼю + різноманітні тестові сценарії |

### Стек спостережуваності (2026)

| Стовп | Стратегія | Ключові метрики |
|--------|---------|-------------|
| Decision Path Mapping | Трасування повного ланцюжка міркувань агентських рішень | Точність використання інструментів, частка досягнення цілей |
| Semantic Drift | Вимірювання відстані ембедінгів між виводами з часом | Cosine similarity відносно семантичного базеліну |
| Cost Observability | Відстеження токенів у реальному часі по агенту/задачі | Токени на вирішену задачу, ROI по агенту |
| Anomaly Detection | Тегування AI-згенерованих змін, кореляція з продакшен-інцидентами | AI-Attributed Change Failure Rate |

**Ключові інструменти:** **Braintrust** (оцінка в продакшен-трейсах), **Arize Phoenix** (open-source виявлення дрифту) та подібні платформи спостережуваності, які підтримують тегування AI-згенерованого коду та поведінкове порівняння.

### Практичні рекомендації

1. **Тегуйте весь AI-згенерований код** у метаданих VCS (повідомлення комітів, мітки PR) для кореляції з інцидентами
2. **Запускайте канаркові деплойменти** для значних AI-згенерованих фіч — направляйте 5% трафіку, порівнюйте з базеліном
3. **Відстежуйте AI Change Failure Rate** поряд зі стандартними DORA-метриками
4. **Моніторте tokens-per-resolved-issue** для виявлення дорогих циклів агентів
5. **Логуйте reasoning traces** для агентських рішень, щоб забезпечити форензичний дебагінг після інцидентів

## 18. Доменно-специфічні агентні патерни

### За межами універсальних агентів для кодування

Універсальні агенти поступаються місцем спеціалізованим «Доменним агентам», які розуміють специфічні обмеження кожного технологічного стеку.

### Матриця доменної архітектури (весна 2026)

| Домен | Ключовий агентний патерн | Специфічні інструменти | Інтеграція з SDD |
|--------|------------------|---------------|-----------------|
| **Frontend/UI** | Пайплайни design-to-code; компонентно-нативна інтеграція | Puck AI, Vercel AI SDK | EARS для поведінки компонентів; Figma MCP для дизайн-токенів |
| **Backend/API** | Spec-to-Controller; агенти міграцій | AutoGen, OpenAI Agents SDK | Специфікації OpenAPI → EARS → згенеровані контролери |
| **Data Engineering** | Трансляція семантичного шару; забезпечення якості | LanceDB, LlamaIndex.js, dbt-AI | EARS для правил якості даних → генерація dbt-тестів |
| **Infrastructure** | Оркестрація Policies-as-Code | Firefly AI, Terraform-AI, Crossplane | EARS для інфраструктурних інваріантів → модулі IaC |
| **Mobile** | UI-рекордери та тріажери | mabl AI Agent | Платформо-специфічний EARS для iOS/Android |

**Акцент на Frontend:** Інструменти design-to-code можуть конвертувати URL-адреси Figma або скріншоти в React/Tailwind код з **високою візуальною точністю** для стандартних компонентів, зберігаючи токени дизайн-системи та точність макету. Однак поточні інструменти досягають **практично нульової функціональної повноти** — вони створюють статичну розмітку без бізнес-логіки, обробки подій чи управління станом. Надайте агенту: бібліотеку компонентів як контекст, поведінкові специфікації (EARS) та Figma MCP для прямого витягування токенів. Розглядайте вивід design-to-code як візуальний каркас, який потребує інженерної роботи для функціональності.

**Акцент на Data Engineering:** Агенти потребують **централізованого семантичного шару** для трансляції бізнес-наміру у запити до бази даних — обходячи вузькі місця BI. EARS-специфікації визначають інваріанти якості даних, які агенти забезпечують через згенеровані dbt-тести.

### Мультимодальні можливості у розробці

Сучасні VLM (vision-language models) можуть інтерпретувати візуальний ввід напряму. Точність суттєво варіюється залежно від складності та контексту:

| Можливість | Зрілість | Випадок використання |
|-----------|----------|----------|
| Figma → React/Tailwind код | Production-ready для простих компонентів | Генерація компонентів |
| Скріншот → код | Добре для стандартних макетів | Відтворення багів, клонування UI |
| ERD → API + міграції | На етапі становлення | Розробка на основі бази даних |
| Діаграма послідовності → реалізація | Експериментально | Оркестрація робочих потоків |
| Дошка → структурована специфікація | Рання стадія | Архітектурні ескізи → специфікації |
| Голос → специфікація | Рання стадія | Архітектор диктує вимоги |

## 19. Стратегії відкату та відновлення після збоїв

### Захисна інженерія для автономних агентів

Галузевий стандарт безпеки агентів базується на **OWASP Top 10 for LLM Applications** та новішому **OWASP Top 10 for Agentic Applications (2026)** — який включає **ASI08: Cascading Failures**, що спеціально адресує поширення збоїв у мультиагентних системах. Команди повинні впроваджувати систематичну обробку збоїв — а не просто «зупиняти агента, якщо він застряг».

### Circuit breakers: жорсткі ліміти

| Поріг | Дія | Обґрунтування |
|-----------|--------|-----------|
| 10–15 циклів міркувань без вирішення | Зупинка, ескалація на людину | Запобігання нескінченним дорогим циклам |
| Бюджет токенів перевищено (>$50/задача) | Пауза, потрібне підтвердження людини | Захист від витрат |
| 3 послідовні невдачі тестів на одній логіці | Відкат до простішого підходу | Запобігання антипатерну «виправлення виправлення» |
| Виявлена деструктивна операція | Потрібне явне підтвердження людини | Бар'єр безпеки |

### Патерн плавної деградації

```
Frontier-модель зазнає невдачі → Повтор з розширеним мисленням
  → Все ще невдача → Відкат до простішої моделі (mid-tier OS-модель)
    → Все ще невдача → Відкат до генерації шаблону/каркасу
      → Все ще невдача → Ручне кодування (людина бере на себе)
```

Кожен крок логується з reasoning traces для post-mortem аналізу.

### Дебагінг AI-коду: форензичний підхід

| Традиційний дебагінг | Дебагінг AI-коду |
|----------------------|-------------------|
| «Чому розробник написав це?» | «Чому агент прийняв таке рішення?» — перевірте reasoning traces |
| Виправити код напряму | Виправити **специфікацію/промпт** → перегенерувати → верифікувати |
| Дебагінг через потік виконання | Дебагінг через **ланцюжок рішень** (який контекст бачив агент?) |
| Одноразове виправлення | Оновити конвенції для запобігання того ж класу помилок |

**Ключовий принцип:** Якщо агент неправильно зрозумів вимогу, виправляйте **специфікацію**, а не код. Ручні правки AI-згенерованого коду створюють «AI-борг» — код, якого ніхто (включаючи майбутніх агентів) не розуміє.

### Відновлення після отруєння контексту

1. **Виявлення**: Якість виводу раптово поіршується або суперечить відомим вимогам
2. **Діагностика**: Перевірте, які файли контексту, відповіді MCP або історія ввели хибні дані
3. **Відновлення**: Очистіть сесію агента, очистіть vector store до стану «останній відомий робочий»
4. **Запобігання**: Тегуйте джерела контексту часовими мітками; впровадьте перевірки на застарілість; відділяйте стабільний контекст (конвенції) від волатильного контексту (дані часу виконання)

### Планування на випадок відмови моделі

| Сценарій | Реагування | Підготовка |
|----------|----------|-------------|
| Основний провайдер недоступний | Маршрутизація на резервного провайдера | Попередньо налаштувати fallback в AI Gateway |
| Усі хмарні моделі недоступні | Перехід на локальні моделі | Підтримувати можливість інференсу on-premise |
| Rate limiting / throttling | Черга за критичністю задач | Управління бюджетом токенів |
| Стрибок цін API | Перевести низькоскладні задачі на дешевші моделі | Конфігурація гібридної маршрутизації |
